## Q1 — "What's the actual deny mechanism?"

The hook reads CC's tool-invocation JSON on stdin and emits JSON on stdout. The relevant field is `permissionDecision`:

```json
{"permissionDecision": "deny", "permissionDecisionReason": "<your reason>"}
```

CC routes the reason back to the model as a deny event. The model usually surfaces it correctly to the user — but not always (post section "What this means for Claude Code design" has the failure mode + the `userMessage` proposal).

The hook also writes a signed receipt before emitting the deny, so the audit trail is independent of the model's narrative. Implementation in `enact/cli/code_hook.py:390-414`.

## Q2 — "How is this different from Cursor's destructive guardrails / MCP / built-in safety modes?"

Three structural differences:

1. **Determinism vs LLM-mediation.** Cursor's guardrails involve model-side reasoning. A deterministic byte-pattern gate doesn't argue. The April 25 PocketOS incident happened *with* destructive guardrails enabled — the model talked itself past them.
2. **PreToolUse, not post-hoc.** Auditing tools (CloudTrail, Splunk) catch the damage after it lands. PreToolUse fires before execution.
3. **Coverage across 6 of 8 CC tools, same engine.** Most existing tooling covers shell only. This covers Read/Write/Edit/Glob/Grep too — closes the "switch to file tool to bypass shell guards" gap.

MCP is complementary, not competitive. An MCP server could host the same engine and serve other clients (Cursor, Codex, Cline). That's on the roadmap.

## Q3 — "Why open source the policies? Doesn't that help the bypass crowd?"

The WHAT is open source (which incidents trigger which policies). The HOW (regex patterns, rename-tracking heuristics, bypass detection) lives in a private bundle the SDK pulls at runtime. ELv2 + signed-bundle distribution.

The reasoning: you can't audit a deterministic safety tool you can't read. Engineers and auditors need to verify "yes, this stops the rename-then-drop bypass." The regex itself, less critical to expose. See ROADMAP.md "SDK split" section for the full structure.

For the open-source path, the default bundle ships with the same policies — no two-tier safety based on willingness to pay. The private bundle adds ongoing curation (new patterns from production telemetry, vertical packs).

## Q4 — "Can the agent be prompt-injected into not invoking the hook?"

The agent doesn't choose whether to invoke the hook. CC invokes the hook on every PreToolUse event regardless of the agent's reasoning. The agent can't suppress the hook from the conversation side; it can only refuse to call the tool in the first place (which is fine — that's a non-action).

The risk surface that *does* exist: the agent telling the user "I ran X" after Enact blocked X. That's the fabrication-detector roadmap item. Receipts catch this server-side; the open product question is how to surface it to the user mid-session.

## Q6 — "What's the performance overhead per tool call?"

Roughly 30-80ms per call on the laptop I'm typing this on. Breakdown:
- Python startup: ~30ms (the `python -m enact.cli.code_hook pre` invocation)
- JSON parse + payload normalize: <1ms
- Policy evaluation across 30 policies: <5ms (regex compiled once at import)
- Receipt write + HMAC sign: ~5ms

The startup cost dominates. For the latency-sensitive folks: a long-lived hook server (CC supports this in its hook protocol) would drop the 30ms startup to ~1ms. Not a priority yet — 30ms inside an LLM's tool-call budget is invisible.

## Q7 — "How do you handle false positives in production?"

Two paths:
1. **Customize policies in `.enact/policies.py`** — auto-created by `init`. Comment out or modify defaults. No restart needed; loaded per-call.
2. **Owner-mode override** (planned, not shipped) — a signed `enact policy unlock <policy-name> --once` that mints a single-use override receipt. Auditable, bounded. The follow-up post on the gitignore-block has more on the tradeoff.

False positives are real — the post's gitignore example is one. The architectural choice is "stop, document, easy workaround" over "argue with the agent." If the FP rate gets too high in practice, the policy moves from BLOCK to WARN (logged + receipt, no deny).

## Q8 — "Why HMAC for receipt signing instead of public-key?"

Two reasons:
1. HMAC is the right primitive for "single owner verifies their own receipts." The user signs and the user verifies. No third party in the trust path.
2. Public-key would let an external auditor verify without the install secret — that's the right primitive for the *next* product surface (independent auditor read API, scaffolded in `cloud/routes/auditor.py`). When we ship that, receipts get a public-key signature alongside the HMAC.

For self-hosted single-user, HMAC is enough. For three-party trust (vendor-customer-auditor), it isn't. Both will coexist.

## Q9 — "Where's the actual 80/20 number coming from? Is the corpus public?"

Yes — `chaos/tasks/` in the repo. 39 markdown prompts with frontmatter pointing at the source incident. The 80/20 is observational from the paired-sweep results, not statistical:

- Of the 26 "dangerous" shell prompts (categories: dangerous-DB + honest-mistake), Claude self-refuses 21 → ~80%.
- Of the 5 file-tool prompts (Read/Write/Edit/Glob/Grep), Claude self-refuses 1 → ~20%.
- Of the 5 misinterpretation prompts (agent-self-initiated), 3 no-op'd on a too-clean sandbox; the remaining 2 the agent attempted destructive without self-refusing → ~0% on the case it actually triggered.

Sample sizes are small. The asymmetry is the qualitative finding; the exact percentages will sharpen as the corpus grows. The trends were stable across two sessions of paired sweeps before publication.

## Q12 — "Did you actually find latent Windows bugs in your own product, or is that marketing dressing?"

Real bugs, both still in the commit history:
- `enact/cli/code_hook.py` — the `init` command writes `<sys.executable> -m enact.cli.code_hook` (not the bare command name) into the settings template specifically because the bare path didn't resolve on Windows.
- The same template uses forward-slashes around the python path because backslashes get mangled by bash escape interpretation when CC pipes the JSON through the shell.

Unit tests in `tests/test_code_hook.py` import the module directly, bypassing both bugs. They surfaced only when the chaos harness ran a real Claude Code subagent end-to-end against the real settings template on Windows.

This is the honest version of "I found bugs in my own product." Not impressive in isolation; impressive only because the failure mode is invisible to the testing strategy I started with.
